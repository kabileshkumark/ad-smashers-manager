## Release Summary

Version 1.0 remains the formal release line. Technical build 1.0.7 is a backward-compatible session scheduling correction after technical build 1.0.6 introduced variable court allocation and weekly recurrence.

The app now records the original court bookings and derives one non-overlapping active-court timeline for cards, fees, peak courts, allocation, and capacity suggestions. For example, 2 courts from 7-9 PM plus 1 court from 7-8 PM is retained as two bookings and presented operationally as 3 courts from 7-8 PM then 2 courts from 8-9 PM.

## Included Changes

- Court Bookings accept adjacent, gapped, overnight, and additive overlapping booking rows within one 24-hour session.
- Original booking rows are the persisted source of truth; the `3 -> 2` active-court timeline is derived by one canonical calculator.
- Court fee uses total court-hours across original bookings without double counting or losing additions.
- Peak simultaneous courts drive booked/planned court count and the automatic capacity suggestion.
- Capacity is editable per session. A manual value remains fixed when bookings change; Reset restores peak courts multiplied by Players per Court.
- Original booking rows, manual capacity, recurrence copies, Firestore conversion, session cards, messages, Dashboard metrics, and allocation share the same domain behavior.
- Legacy `courtSlots` sessions remain readable and retain stored capacity. They convert to `courtBookings` only when edited.
- Booking and capacity changes use the existing active cash, Advance, and Credit financial-basis guard.
- The Requirement Ambiguity Gate was added to Coding Best Practices and Jira/Codex templates so material unknowns require owner decisions before implementation.
