## Release Summary

Version 1.0 remains the formal product release. Technical build 1.0.10 corrects the session financial-mutation guard without changing the Firebase schema or payment ownership model.

Automatically derived Advance and Credit are recalculable ledger coverage, not immutable receipt history. A user can therefore correct court bookings, time, capacity, cost, or rate when that is the only coverage on the session. Recorded cash, stored receipt amounts, and active transaction allocations remain protected until the corresponding receipt is reversed or deleted.

## Included Changes

- Separates recorded session payments from automatically derived Advance and Credit coverage.
- Allows session financial-basis corrections when only derived coverage exists.
- Recalculates coverage chronologically after the corrected charge is synchronized.
- Retains the existing lock for recorded cash and active transaction allocations.
- Retains roster, retained-history, session-deletion, and payment-ownership controls.
- Presents a clear confirmation when the save recalculates Advance and Credit.
- Adds regression coverage for own Advance, payment-group Credit, recorded payment, and transaction-allocation paths.
- Preserves the production export outside Git and deploys Firebase Hosting only.
